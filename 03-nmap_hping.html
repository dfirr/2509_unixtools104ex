3  パケットの操作（nmap, hping）

公開

2026年1月20日

更新日

2026年3月5日

3.1 目的

この章の目的は、次の通りです。

  • nmapによるネットワークスキャンの基本が理解できるようになること。
  • nmapのscript機能を使って、対象ホストの追加情報が得られるようになること。
  • hping3を使って、より低いレベルでパケットを操作できるようになること。
注意

この章でスキャン対象とするホストは、すべてホスト内のDockerの閉域網内で実施します。インターネットや社内ネットワーク上のホストに対してはスキャンしません。

自分たちが管理していないホストに対する無断スキャンは、思わぬ障害や法令違反につながる恐れがあります。必ず、講義用に準備された環境のみを対象にしてください。

3.2 事前準備（git, Docker）

上記の理由から、WSLにおいて、gitとDocker関連ツールをインストールします。

sudo apt update
sudo apt install -y git docker.io docker-compose-v2

Dockerをsudoなしで使えるように、ユーザーをdockerグループに追加します。

sudo usermod -aG docker $USER

この変更は、ログインし直すと反映されます。WSLターミナルを閉じて開き直すか、次のコマンドで現在のシェルに反映してください。

newgrp docker

以下で動作確認します。

git --version
docker --version
docker compose version

3.2.1 演習ファイルの取得

GitHubから演習ファイルを取得します。以下ではcasesというディレクトリに保存していますが、好きに変更して構いません。

mkdir -p ~/cases
cd ~/cases
git clone https://github.com/dfirr/2509_unixtools104ex.git
cd 2509_unixtools104ex/ch3-lab

3.2.2 演習環境の起動

次のコマンドで演習環境を起動します。

docker compose up -d --build

起動するDockerコンテナは、以下の3つです。

  • ch3-scanner: nmap, hping3, tcpdump, nc, curlを実行するホスト（10.250.250.10）
  • ch3-target-web: 80/tcpを待ち受けるホスト（10.250.250.20）
  • ch3-target-closed: 待ち受けポートを持たないホスト（10.250.250.30）

この演習では、Dockerネットワークとして10.250.250.0/24を使用します。

正常に起動できたかを、次のようにして確認します。

docker compose ps

3.2.3 scannerで使えるコマンド確認

scannerコンテナに入り、この章で使うコマンドを確認します。

docker exec -it ch3-scanner bash
which nmap hping3 tcpdump nc curl

確認できたら、以後のコマンドはscannerコンテナ内で実行します。

3.3 nmap

3.3.1 nmapの基本

nmapは、ネットワーク探索とポートスキャンを行うためのツールです。基本的な書式は次の通りです。

nmap [オプション] 対象

最初に、対象ネットワークの中でどのホストが応答するかを確認します。-snオプションを使うと、ポートスキャンを行わず、ホスト発見だけを行います。

nmap -sn 10.250.250.0/24

ここでは、10.250.250.20と10.250.250.30とが応答することを確認してください。

3.3.2 ホスト発見スキャン

対象の書き方には、いくつかの方法があります。代表的なものは次の2つです。

  • CIDR表記
  • 範囲表記
nmap -sn 10.250.250.0/24
nmap -sn 10.250.250.1-254
ノート

-snは「Ping Scan」と呼ばれますが、ICMP Echo Requestを送るとは限りません。

nmapは、明示的な指定がないとき、既定で複数の発見プローブを使います。具体的には、ICMP Echo Request、ICMP Timestamp Request、TCP SYN（443/tcp）、TCP ACK（80/tcp）です。

ただし、今回のように同一セグメントに対してIPv4で実行するときには、ARPが優先されます。

重要

nmapのスキャンは、実行権限によっても挙動が変わります。root権限があるとraw packetを用いた発見プローブを利用できますが、non-rootではconnect()スキャンしかできません。よって、nmapはroot権限で実行するようにしてください。

Problem問題

ここで、1つ疑問が生じます。non-rootではICMP Echo Requestを送れないはずなのに、Linuxでは一般ユーザーでもpingコマンドを実行できます。これはなぜでしょうか？

ヒントヒント

which pingやgetcapコマンドを使って調べてください。

ヒント解答

Ubuntu 24.04 LTS では、pingコマンド自体にcapabilityが付与されています。capabilityとは、root権限を丸ごと与えるのではなく、特定の操作だけを許可する仕組みです。

昔は、この種のコマンドを一般ユーザーに使わせるには、setuid rootのようにroot権限で動かす方法が使われていました。それに対してcapabilityは、必要な権限だけを細かく与える仕組みです。

次のように確認できます。

which ping
getcap /usr/bin/ping

典型的には、次のように表示されます。

/usr/bin/ping
/usr/bin/ping cap_net_raw=ep

ここでcap_net_rawは、raw socketを使うための権限です。pingコマンドにはこの権限だけが与えられているため、一般ユーザーが実行してもICMP Echo Requestを送信できます。

一方、nmapに同じ権限が与えられているとは限りません。そのため、一般ユーザーとして実行したときには、pingは使えても、nmapではraw packetを用いたホスト発見が制限されます。

Problem問題

nmap -sn 10.250.250.0/24の結果から、応答したホスト数を確認してください。target-webとtarget-closedは見つかるでしょうか？

ヒント解答

この演習環境では、4ホストが応答します。筆者の環境では、次のようになりました。

Nmap scan report for 10.250.250.1
Host is up

Nmap scan report for d850413e063a (10.250.250.10)
Host is up

Nmap scan report for ch3-target-web.ch3-labnet (10.250.250.20)
Host is up

Nmap scan report for ch3-target-closed.ch3-labnet (10.250.250.30)
Host is up

10.250.250.20がtarget-web、10.250.250.30がtarget-closedです。これらに加えて、scanner自身（10.250.250.10）と、Dockerネットワークのゲートウェイ側である10.250.250.1も応答します。

したがって、target-webとtarget-closedとは、どちらも見つかります。

3.3.3 ポートスキャン

ホストの存在が確認できたら、そのホストのどのポートが開いているかを調べます。

nmap target-webのようにポート番号を指定しない場合、nmapはデフォルトで全65535ポートを調べるのではなく、使用頻度の高い1000個のTCPポートを対象にします。

-pで対象ポートを指定します。まず、80/tcpだけを対象にしてみます。

nmap -p 80 target-web
Starting Nmap 7.94SVN ( https://nmap.org ) at 2026-03-01 13:56 UTC
Nmap scan report for target-web (10.250.250.20)
Host is up (0.000061s latency).
rDNS record for 10.250.250.20: ch3-target-web.ch3-labnet

PORT   STATE SERVICE
80/tcp open  http
MAC Address: CE:06:12:69:15:72 (Unknown)

Nmap done: 1 IP address (1 host up) scanned in 0.23 seconds

出力では、次の列に注目してください。

  • PORT: ポート番号とプロトコル
  • STATE: ポート状態（open/closed/filtered）
  • SERVICE: 推定されたサービス名

root権限でnmapを実行したとき、デフォルトのTCPスキャンは-sSです。-sSはSYN scanであり、TCPの3ウェイハンドシェイクを最後まで完了させずに状態を推定します。

nmap -sS -p 80 target-web

一方、non-rootではraw packetを使えないため、-sSは使えません。その場合のデフォルトは-sTです。-sTはTCP connect scanであり、OSの通常ソケットAPIを使って接続を試みます。

nmap -sT -p 80 target-web
ノート

nmapがroot権限で-sSをデフォルトとして使うのは、スキャン用途に向いているからです。-sTではOSに完全なTCP接続を張らせますが、-sSではSYNとそれに対する応答だけを見て、接続確立前の段階で状態を判定できます。

そのため、-sSのほうが効率的であり、相手側にも完全な接続としては見えにくい、という利点があります。

3.3.4 状態推定（open/closed/filtered）

nmapの出力では、ポート状態としてopen, closed, filteredが表示されます。target-webとtarget-closedを比較し、結果の違いを確認してください。

nmap -p 80,443 target-web target-closed

大まかな意味は次の通りです。

  • open: 対象ポートで待ち受けしているプロセスがある
  • closed: 対象ホストには到達できるが、待ち受けはない
  • filtered: 無応答、またはICMP Destination Unreachable（type 3）の一部コードにより、openかclosedかを判定できない
    • 例: code 1（Host Unreachable）、code 2（Protocol Unreachable）、code 3（Port Unreachable: TCP系）、code 9/10/13（Administratively Prohibited）

3.3.5 ACKスキャン（-sA）

-sAはACKパケットを使って、フィルタリングの有無を推定する手法です。

nmap -sA -p 80 target-web
nmap -sA -p 80 target-closed
ヒント観察のポイント

今回の演習環境では、target-webとtarget-closedとの両方でunfilteredになるはずです。これは、フィルタされていないことを意味しますが、ポートが開いていることを意味するわけではありません。

-sSと-sAとでは、得られる情報の種類が異なります。-sSはポート開閉の推定に、-sAはフィルタの有無の推定に向いています。

ノート

もし途中にファイアウォール等があり、応答が返ってこなければ、filteredと表示されます。今回の演習ラボでは、その状況は意図的には作っていません。

3.3.6 バージョン検出・OS推定

-sVでサービスバージョン検出を行います。これは、アプリケーションが返す応答の内容から、サービスの種類やバージョンを推定する機能です。

nmap -sV -p 80 target-web
Starting Nmap 7.94SVN ( https://nmap.org ) at 2026-03-01 14:08 UTC
Nmap scan report for target-web (10.250.250.20)
Host is up (0.00010s latency).
rDNS record for 10.250.250.20: ch3-target-web.ch3-labnet

PORT   STATE SERVICE VERSION
80/tcp open  http    nginx 1.29.5
MAC Address: CE:06:12:69:15:72 (Unknown)

Service detection performed. Please report any incorrect results at https://nmap.org/submit/ .
Nmap done: 1 IP address (1 host up) scanned in 6.47 seconds

SERVICE列だけでなく、VERSION列にも注目してください。今回の例では、httpサービスがnginx 1.29.5であることまで推定できています。

なお、-sVを付けないときのSERVICE列は、ポート番号との対応表に基づくものであり、それは実際の応答を調べた結果ではありません。-sVを使うことで、実際の応答内容からサービスの種類やバージョンをより正確に推定できます。なお、対応表は/usr/share/nmap/nmap-servicesに入っています。

-OはOS推定です。TCP/IPの応答の特徴（TTL、ウィンドウサイズ、フラグ応答など）からOSを推定します。

nmap -O target-web
Starting Nmap 7.94SVN ( https://nmap.org ) at 2026-03-01 14:08 UTC
Nmap scan report for target-web (10.250.250.20)
Host is up (0.000095s latency).
rDNS record for 10.250.250.20: ch3-target-web.ch3-labnet
Not shown: 999 closed tcp ports (reset)
PORT   STATE SERVICE
80/tcp open  http
MAC Address: CE:06:12:69:15:72 (Unknown)
Device type: general purpose
Running: Linux 4.X|5.X
OS CPE: cpe:/o:linux:linux_kernel:4 cpe:/o:linux:linux_kernel:5
OS details: Linux 4.15 - 5.8
Network Distance: 1 hop

OS detection performed. Please report any incorrect results at https://nmap.org/submit/ .
Nmap done: 1 IP address (1 host up) scanned in 1.52 seconds

OS推定はTCP/IPスタックの指紋を使うため、NATやファイアウォール、ポートフォワーディングなどで応答が変わると結果が乱れることがあります。可能なら対象と近いネットワークから測定するほうが精度が上がります。

-Pnを使うと、事前のホスト発見を省略してスキャンします。

nmap -Pn -p 80 target-web
ノート

-Pnは、ホスト発見用のプローブに応答しない相手に対しても、「ホストは起動している」と仮定してポートスキャンを続行したいときに使います。

たとえば、あるホストが非標準ポート8080/tcpだけで待ち受けていて、既定のホスト発見プローブには反応しないとします。このとき、-Pnを付けないと「host seems down」と判断され、ポートスキャン自体に進まないことがあります。

3.3.7 スクリプトスキャン

nmapにはNSE（Nmap Scripting Engine）があり、ポートが開いていることだけでなく、その先のアプリケーション層を追加調査できます。今回は、HTTP系のスクリプトだけを取り上げます。

HTTP系のスクリプトは多数あります。まず、どのようなものがあるか一覧を見てみます。

ls /usr/share/nmap/scripts/http-* | head

ここでは、次の3つを使います。

  • http-title: HTMLのtitle要素を取得する
  • http-headers: HTTPレスポンスヘッダーを取得する
  • http-server-header: Serverヘッダーを取り出す

まず、HTTP系スクリプトを使ってWebサーバーの情報を収集します。

nmap -p 80 --script http-title,http-headers,http-server-header target-web
Starting Nmap 7.94SVN ( https://nmap.org ) at 2026-03-01 14:34 UTC
Nmap scan report for target-web (10.250.250.20)
Host is up (0.000056s latency).
rDNS record for 10.250.250.20: ch3-target-web.ch3-labnet

PORT   STATE SERVICE
80/tcp open  http
|_http-server-header: nginx/1.29.5
|_http-title: Ch3 Lab Target Web
| http-headers:
|   Server: nginx/1.29.5
|   Date: Sun, 01 Mar 2026 14:34:04 GMT
|   Content-Type: text/html
|   Content-Length: 287
|   Last-Modified: Wed, 25 Feb 2026 14:25:11 GMT
|   Connection: close
|   ETag: "699f0647-11f"
|   Accept-Ranges: bytes
|
|_  (Request type: HEAD)
MAC Address: E2:23:C8:1F:E3:7C (Unknown)

Nmap done: 1 IP address (1 host up) scanned in 0.37 seconds

User-Agentを変えたいときには、--script-argsでhttp.useragentを指定します。

nmap -p 80 --script http-title --script-args http.useragent='Mozilla/5.0' target-web

このように、HTTP系スクリプトでもcurlと同様に、リクエストの細部を調整できます。

Problem問題

公開されているパスを推測によって探索してください。どのHTTP系スクリプトを使えばよいでしょうか。

ヒントヒント

先ほど一覧表示したHTTP系スクリプトの中には、よく使われるパス候補を調べるものがあります。

ヒント解答

http-enumを使います。なお、http-enumは、よく使われるパス候補に対してHTTPリクエストを送り、応答コードから存在を推定します。辞書にもとづく推測探索なので、何でも見つけられるわけではありません。

nmap -p 80 --script http-enum target-web
Starting Nmap 7.94SVN ( https://nmap.org ) at 2026-03-01 14:34 UTC
Nmap scan report for target-web (10.250.250.20)
Host is up (0.000043s latency).
rDNS record for 10.250.250.20: ch3-target-web.ch3-labnet

PORT   STATE SERVICE
80/tcp open  http
| http-enum:
|   /admin.html: Possible admin folder
|_  /robots.txt: Robots file
MAC Address: E2:23:C8:1F:E3:7C (Unknown)

Nmap done: 1 IP address (1 host up) scanned in 2.00 seconds
Problem問題

http-enumで検出された候補を、curlで実際に確認します。

curl -i http://target-web/admin.html

/admin.htmlは、target-webのトップページからリンクされているでしょうか。また、攻撃者がこのページを発見できる理由を説明してください。

ヒントヒント

curl -i http://target-web/robots.txt を実行してみてください。

ヒント解答

たとえば、次のように確認できます。

curl -s http://target-web/robots.txt
User-agent: *
Disallow: /admin.html

リンクが無くても、推測しやすいURLや列挙スクリプトの辞書探索により発見される可能性があります。robots.txtの記述も手がかりになります。

3.3.8 結果のファイル出力

nmapの結果は、画面に表示するだけでなく、ファイルにも保存できます。通常のテキスト形式で保存するには、-oNを使います。

nmap -sV -p 80 target-web -oN result.txt
cat result.txt

複数の形式でまとめて保存したいときには、-oAが便利です。

nmap -sV -p 80 target-web -oA result

このとき、次の3つのファイルが生成されます。

  • result.nmap: 通常のテキスト形式
  • result.xml: XML形式
  • result.gnmap: grepしやすい形式

3.4 hping3

3.4.1 hping3入門

hping3は、任意のフラグを付けたTCPパケットなどを手動で送信できるツールです。nmapより低レベルの実験ができます。

nmapが「どのような情報を得たいか」を指定する道具だとすれば、hping3は「どのようなパケットを送るか」を自分で決める道具です。そのぶん、手間は増えますが、TCP/IPの挙動をより直接的に確かめられます。

ノート

もっと手軽な実務例として、社内LANでICMP Echo Requestがブロックされていてpingでは疎通確認できない場合でも、インターネット上のWebサイトに対して443/tcpへSYNを送り、応答があるかどうかで死活確認する、という使い方があります。

このように、hping3は「任意のプロトコルで疎通確認したい」ときにも役立ちます。

以下のコマンドは、WSLのUbuntuそのものではなく、scannerコンテナ内で実行します。tcpdumpとhping3とを同時に観察したいので、WSLでターミナルを2つ用意し、それぞれで次のコマンドを実行してscannerコンテナに入ってください。

docker exec -it ch3-scanner bash

まずは、80/tcpに対してSYNパケットを送信します。target-webの80/tcpは開いているので、SYNに対してSYN/ACKが返るはずです。

hping3 -S -p 80 -c 3 target-web

-SはSYNフラグ、-pは宛先ポート、-cは送信回数です。

HPING target-web (eth0 10.250.250.20): S set, 40 headers + 0 data bytes
len=44 ip=10.250.250.20 ttl=64 DF id=0 sport=80 flags=SA seq=0 win=64240 rtt=6.3 ms

上記のflags=SAは、SYN/ACKが返ってきたことを意味します。これは、相手がTCP接続の確立に応じようとしていることを表します。

ノート

hping3では、自分が作ったSYNに対して相手からSYN/ACKが返ったあと、ローカルのカーネルがRSTを返すことがあります。これは、そのSYNに対応する通常のソケットが存在しないためです。

したがって、tcpdumpで観察するとSYN -> SYN/ACK -> RSTのように見えることがあります。

たとえば、tcpdumpでは次のように観察できます。

14:56:55.081980 IP 10.250.250.10.40000 > 10.250.250.20.80: Flags [S], seq 1926046387, win 512, length 0
14:56:55.082035 IP 10.250.250.20.80 > 10.250.250.10.40000: Flags [S.], seq 3080780592, ack 1926046388, win 64240, options [mss 1460], length 0
14:56:55.082043 IP 10.250.250.10.40000 > 10.250.250.20.80: Flags [R], seq 1926046388, win 0, length 0

3.4.2 hping3演習（SYN/ACK/フラグ操作）

次に、SYN以外のフラグでも挙動を確認します。

まず、ACKだけを送ってみます。既存のTCP接続が無い状態でACKだけを受け取った相手は、通常はRSTを返します。

hping3 -A -p 80 -c 3 target-web

tcpdumpでは、たとえば次のように見えます。

14:58:13.947661 IP 10.250.250.10.40001 > 10.250.250.20.80: Flags [.], ack 567889971, win 512, length 0
14:58:13.947704 IP 10.250.250.20.80 > 10.250.250.10.40001: Flags [R], seq 567889971, win 0, length 0

次に、FINだけを送ってみます。これも今回の環境ではRSTが返るはずです。

hping3 -F -p 80 -c 3 target-web

送信したFINパケット自体は、tcpdumpでは次のように見えます。

14:59:17.328783 IP 10.250.250.10.40002 > 10.250.250.20.80: Flags [F], seq 1677856718, win 512, length 0

今回の演習で注目したい対応関係は、次の通りです。

送るフラグ 予想される応答 意味
SYN SYN/ACK ポートが開いており、接続確立に応じようとしている
ACK RST 既存接続が無いため、相手が異常なパケットとみなしている
FIN RST 今回の環境では、接続状態が無いため終了要求として扱えない

1つ目のターミナルでは、scannerコンテナ内で次を実行し、パケットを観察してください。

tcpdump -n -i eth0 'host target-web and tcp'

2つ目のターミナルでも、scannerコンテナに入った上で、hping3 -S ...、hping3 -A ...、hping3 -F ...を順に実行します。

tcpdumpでは、こちらが送ったフラグだけでなく、相手が返したフラグ（S., Rなど）に注目してください。

3.5 追加課題

3.5.1 非標準ポートでの待ち受け

target-webコンテナは、80/tcp以外の別のTCPポートで何らかのサービスを待ち受けています。ただし、そのポートはnmapでデフォルトで使用される1000ポートには含まれていません。

Problem問題

nmap target-webでは見つからない、もう1つの開いているTCPポートを見つけてください。

ヒントヒント

より広い範囲を調べるには、-pでポート範囲を明示してください。

ヒント解答

たとえば、次のようにすると見つけられます。

nmap -p 1-20000 target-web

18000/tcpが見つかります。

Problem問題

見つけたポートで待ち受けているサービスの種類を調べてください。

ヒントヒント

nmap -sVを使います。

ヒント解答

たとえば、次のように確認できます。

nmap -sV -p 18000 target-web
curl -I http://target-web:18000/

nmap -sVではHTTP系のサービスであることが推定されます。さらにcurlを実行すると、HTTP/1.1 200 OKが返るため、このポートで待ち受けているのがHTTPサーバーだと分かります。

3.5.2 意図せざるファイルの露出

Webサーバーの設定によっては、認証で保護されたディレクトリがある一方で、認証情報ファイルそのものが誤って公開されてしまうことがあります。

Apacheでは.htaccessと.htpasswdの組み合わせで起きやすい事故として知られています。ここでは、その種の事故を模した状態を、直前に見つけた18000/tcp側に用意してあります。

Problem問題

target-webの18000/tcpに対してHTTP系のスクリプトスキャンを行い、認証で保護されたディレクトリと、誤って公開されている認証情報ファイルを見つけてください。

ヒントヒント

公開パスの列挙には、http-enumが使えます。

ヒント解答

たとえば、次のように実行できます。

nmap -sV -p 18000 --script http-enum target-web

結果には、次のような行が現れます。

| http-enum:
|   /.htpasswd: Incorrect permissions on .htaccess or .htpasswd files
|_  /secret/: Potentially interesting folder (401 Unauthorized)

これにより、/secret/が認証で保護されている一方で、/.htpasswdが公開されていることが分かります。これは、認証情報ファイルが外部から取得できてしまうという、重大なセキュリティ上の不備です。

Problem問題

/secret/では、どのHTTP認証方式が使われているか確認してください。

ヒントヒント

http-authを使い、調査対象のパスを指定します。

ヒント解答

たとえば、次のように確認できます。

nmap -sV -p 18000 --script http-auth --script-args http-auth.path=/secret/ target-web

結果は次のようになります。

| http-auth:
| HTTP/1.1 401 Unauthorized
|_  Basic realm=Restricted

このため、/secret/はHTTP Basic認証で保護されていると分かります。

Problem問題

/.htpasswdを取得し、パスワードを推測してください。その後、実際に/secret/へアクセスして確かめてください。

ヒントヒント

ファイルの取得にはcurlを使えます。最後はcurl -uで認証付きアクセスを試します。

ヒント解答

まず、/.htpasswdを取得します。

curl -s http://target-web:18000/.htpasswd

結果は次の通りです。

student:$apr1$cwQzNMVI$qerkxPFGowaHaOyoTHuI1.

このハッシュをHashcatなどのパスワードクラッキングツールで解析すると、そのパスワードがPasswordだと判明します。そこで、次のように認証情報付きでアクセスします。

curl -i -u student:Password http://target-web:18000/secret/

成功すると、HTTP/1.1 200 OKが返ります。このケースは、認証情報ファイルの露出によって、保護されたディレクトリへアクセスできてしまうことが確認できる一例です。
